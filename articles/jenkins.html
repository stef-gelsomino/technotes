Jenkins Continuous Integration

Einleitung

Jenkins ist hauptsächlich einen build Server und kann mit plugins integriert werden. Die Website ist http://jenkins-ci.org

Im Besonderen:

  • Der Entwickler puscht die Änderungen
  • Jenkins kriegt die letzte Source Version und generiert einen Output (z.B. ein Artefakt)
  • Jenkins kann einen report zum Entwickler schicken (z.B. ein E-Mail, SMS ...)

Installierung

  • Voraussetzung

    • Java mindenstens 1.5 Version
    • Java in PATH setzen
    • einen Web Server installieren (z.B. Tomcat, JBoss)
  • Heruntenladen das war file vor Jenkins Website ("Download Jenkins" -> "Release" tab) und lauft es von Terminal $ java -jar jenkins.war
    ---
    Der Server läuft auf: localhost:8080

    Laufen auf einen andere Port oder mit https

    $ java -jar jenkins.war --httpPort=8081 $ java -jar jenkins.war --httpPort=-1 --httpsPort=8081
    ---
    Der Server läuft auf: localhost:8081

    Oder einen Admin User setzen

    Wann der Server läuft, den Arbeitsverzeichnis (working directory) Pfad kann im Log datei gefunden werden: Jenkins home directory: C:\Users\John\.jenkins

Stichwörter:
  • xUnit Testing
  • Class Under Test
  • Method Under Test
  • Test Fixture (Setup von Daten, die im Test verwendet werden)
Mythen (die falsch sind):
  • verdoppelte Coding-Effort
  • Alle Tests müssen bereits erledigt sein, vor die Funktionalitätscode geschrieben wird: das ist einfach eine Fehlinterpretation von Test-First-Development! Man soll nur einfach den Red/Green/Refactor Zyklus folgen
Testing Frameworks (JVM Welt):
  • Mokckito: Mocking von Abhängigkeiten
  • DBUnit: Fixtures von SQL Datenbanken
H2 Datenbank:
  • relational Datenbank in Java
  • in/out-of-process Ausführung (gut für Tests)
  • das h2 Verzeichnis herunterladen und unzippen in /opt Ordner
